## 目标
- 提供稳定、可扩展、易维护的用户体系（注册/登录、鉴权、会话、权限、审计、风控），适配未来功能扩展与可能的整体重构。
- 与现有学习与评估业务解耦，分层清晰，支持持续交付与监控。

## 推荐技术栈（首选）
- 后端框架：NestJS（TypeScript）
  - 理由：成熟生态、强约束的分层结构、内置良好的 DI 与模块化、官方文档完善、团队规模友好。
- 数据库：PostgreSQL
  - 理由：事务一致性强、JSON/全文搜索能力、社区与云服务支持度高。
- ORM：Prisma
  - 理由：开发体验优秀、类型安全、迁移机制稳定、与 TS 深度集成。
- 缓存/会话/限流：Redis
  - 理由：高性能 KV，适合存储短期会话、黑名单、验证码、限流计数等。
- 鉴权策略：
  - 密码登录（Argon2 哈希）+ 邮件验证码（Resend/SendGrid）
  - JWT（短期 Access Token）+ Refresh Token（持久化/轮换，绑定设备指纹与 IP 近端检查）
  - 可选：Session Cookie（HttpOnly/SameSite=strict）用于 Web 端；移动端使用 JWT。
- 校验与安全：
  - DTO 校验：class-validator + class-transformer
  - 输入校验与反序列化严格化，统一错误码
  - CSRF（如使用 Cookie 会话）、XSS 防护（HTTP 头与前端编码）、速率限制（基于 Redis）
- 日志与审计：pino + OpenTelemetry（可选），审计表记录关键安全事件（登录失败、密码修改、令牌刷新、API Key 使用）
- 配置与部署：
  - Docker Compose（`api`、`postgres`、`redis`）+ `.env` 管理机密
  - CI/CD（GitHub Actions）集成测试与迁移自动化
- 前端集成：
  - 维持 React（或可迁 Next.js 作为 SSR/路由框架）
  - 数据请求：TanStack Query + Axios/Fetch；统一拦截器处理 401/刷新令牌

## 备选技术栈（根据团队偏好）
- Python + FastAPI + SQLAlchemy + Postgres + Redis
  - 优点：语义清晰、生态强；缺点：类型约束略弱，团队 TS 经验需转移
- Go + Chi/Fiber + GORM + Postgres + Redis
  - 优点：性能与资源占用出色；缺点：开发效率与迭代速度相对较慢
- 完整全栈：Next.js（App Router）+ NextAuth（结合自定义凭证）+ Prisma + Postgres
  - 优点：端到端统一；缺点：服务端路由与页面耦合，适合产品导向但后端独立性略弱

## 领域模型（核心表）
- `User`：id、email（唯一）、phone（可选唯一）、password_hash（Argon2）、status、created_at
- `UserProfile`：user_id、name、avatar_url、education_level（与学习业务对齐）
- `AuthProvider`：user_id、provider（email/password、oauth）、provider_uid
- `RefreshToken`：user_id、token_hash、expires_at、revoked_at、device_info、ip_last
- `EmailVerification`：user_id、code_hash、expires_at、consumed
- `PasswordReset`：user_id、code_hash、expires_at、consumed
- `ApiKey`（可选，用于第三方服务调用审计）：name、key_hash、scopes、created_at、revoked_at
- 学习业务对接：`WordMastery`、`LearningSession`（与现有本地存储结构映射，支持云端同步）

## 接口契约（示例）
- `POST /auth/register`：email + password + code（验证码）
- `POST /auth/login`：email + password
- `POST /auth/refresh`：刷新令牌换取新 access
- `POST /auth/logout`：撤销刷新令牌
- `GET /me`：当前登录用户信息
- `PATCH /me`：更新资料
- 风控：所有登录相关接口加入速率限制与审计日志

## 鉴权与会话流程
1. 注册：
   - 提交邮箱与密码（前端强校验），服务端生成验证码并邮件发送，入库 `EmailVerification`
   - 验证通过后创建用户、写入 `User` 与 `AuthProvider`
2. 登录：
   - 验证密码（Argon2），签发短期 `access_token` 与持久化 `refresh_token`（入库并绑定设备）
   - Web 端可使用 HttpOnly Cookie 承载 `access_token`（或 Bearer 头）
3. 刷新：
   - 校验刷新令牌是否撤销与过期；轮换策略（旧刷新令牌失效，发新令牌）
4. 登出：
   - 撤销用户当前设备刷新令牌；可选“全部设备退出”

## 安全与合规
- 密码：Argon2id（并设置合理的 `time_cost`/`memory_cost`）
- Tokens：签名秘钥滚动、JTI 与黑名单策略、设备指纹近端检查
- 速率限制：IP + 用户维度，基于 Redis 的漏桶/令牌桶
- 审计：关键事件入库，可检索与告警（失败登录阈值触发告警）
- 隐私：遵守本地法律与最佳实践（数据最小化、加密静态存储、删除/导出请求流程）

## 与现有业务的结合
- 保持 `/api` 路径风格不变；将学习相关端点置于受鉴权保护的路由（例如 `GET /learning/mastery`）
- 将现有前端的本地 `localStorage` 同步至后端：
  - 首次登录后批量上报本地掌握项
  - 后续增量同步；提供复盘与跨设备共享

## 选择结论
- 首选方案：NestJS + Postgres + Prisma + Redis + JWT/Refresh + Argon2，整体在生产中成熟稳定、维护成本低、扩展便利。
- 若团队更偏 Python 或 Go，可选择备选方案；但就“长期稳定 + 交付效率”综合考虑，首选方案更均衡。
