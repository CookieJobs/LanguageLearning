## 现状梳理
- 前端：Vite + React 在仓库根目录，开发脚本 `vite`，路由位于 `App.tsx:390-396` 与 `react-router-dom` 导入 `App.tsx:11`。
- 后端一（业务 API）：NestJS + Prisma + Redis 位于 `api/`，入口 `api/src/main.ts:11-16`，全局前缀 `api/src/main.ts:12`，认证与学习模块在 `api/src/modules/*`，示例路由 `auth.controller.ts:31-49`、`learning.controller.ts:17-40,42-62,64-79`。
- 后端二（LLM服务）：Express 服务位于 `server/`，暴露 `/api/health`、`/api/words`、`/api/evaluate`，端口 `server/src/server.ts:55-56`。其功能与 `api/learning` 中的生成与评估存在部分重叠。
- 前端到后端代理：`vite.config.ts:12-16` 将 `/api` 代理到 `http://localhost:5500`。
- 部署资产：`deploy/` 下含前端、后端独立 Dockerfile 与 NGINX 配置，根目录 `docker-compose.yml:17-31` 启动 `api`、`postgres`、`redis`。

## 架构目标
- 标准化为“单前端 + 单后端（NestJS）”的清晰分层；将 LLM 相关接口统一进 `api` 的 `learning/ai` 子模块，或保留为独立微服务但通过反向代理提供规范前缀（如 `/llm`）。
- 统一端口与环境变量命名，前端固定 `3000`，后端固定 `5500`（或一致调整为 `5000`，与代理同步）。
- 所有浏览器请求只经过一个入口前缀（`/api` 或 `/llm`），避免路径重叠和歧义。

## 后端整合方案（推荐）
- 在 `api/` 中新增 `ai` 子模块，将 `server/src/geminiService.ts` 与 `server/src/server.ts` 的 `/api/words`、`/api/evaluate` 能力迁移为 `api/learning` 下的服务/控制器方法，保持现有 DTO 与认证（参考 `learning.controller.ts:17-40,42-62`）。
- 统一使用 `DeepSeek` 或 `Gemini` 的调用封装，配置在 `api/src/modules/learning/*` 或新建 `ai.service.ts`，读取密钥自 `.env`（当前 `learning.controller.ts:45-47` 已读取 DeepSeek）。
- 保留 Prisma 的学习记录接口（`/api/learning/mastery*`），并确保评估成功后写库逻辑不变。
- 删除或停用 `server/` 目录，避免与 `api` 的 `/api/*` 路由前缀冲突；若需保留，改用独立前缀 `/llm/*` 并在 NGINX/Vite 代理单独配置。

## 前端对接与代理
- 继续走 `fetch('/api/...')`（参考 `services/geminiService.ts:4,24-48,50-59,61-76,78-85,87-103`），仅在接口合并后校对路径与响应结构。
- 如果保留独立 LLM 微服务，则前端改为调用 `/llm/...` 并在 `vite.config.ts` 增加对应代理，生产环境由 NGINX 反向代理。

## 端口与环境变量统一
- 后端固定监听 `5500` 并与 `vite.config.ts:12-16` 保持一致；或统一改为 `5000` 并同步调整代理与 `docker-compose`（当前 `docker-compose.yml:21,28` 与 `main.ts:14` 存在 5000/5500 不一致）。
- `.env` 中规范化密钥：`DEEPSEEK_API_KEY`、`DEEPSEEK_MODEL`、数据库与 Redis 连接；前端不再通过 `define` 暴露任何服务端密钥（`vite.config.ts:19-22` 应改为仅在后端使用）。

## 本地开发流程
- 启动数据库与缓存：`docker-compose up -d postgres redis`。
- 启动后端：在 `api/` 运行 `npm run dev`（NestJS，端口 `5500`）。
- 启动前端：在根目录运行 `npm run dev`（Vite，端口 `3000`）。
- 通过前端登录与学习流程验证词表、评估与掌握记录全链路。

## 生产部署与反向代理
- 使用 `deploy/frontend/Dockerfile` 与 `deploy/api/Dockerfile` 构建镜像，经 `deploy/nginx/nginx.conf` 统一反代到 `/api`（以及可选 `/llm`）。
- 校准容器端口与健康检查，确保与 `docker-compose.yml` 映射一致。

## 风险与兼容注意
- 接口合并时，统一响应结构与错误码，前端类型推断需同步更新。
- 谨慎处理密钥泄露风险：不在前端打包任何 API 密钥；后端通过环境注入。
- 如需保留 `server/` 微服务，确保路径前缀不与 `api` 重叠，并在代理层做清晰分流。
