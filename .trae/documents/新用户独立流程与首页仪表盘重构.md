## 期望效果
- 新用户首次登录后进入“新用户流程”页面，完成学段选择并持久化。
- 完成选择后进入首页仪表盘，显示“开始学习”“连续打卡”“掌握单词”入口。
- 之后再次登录直接进入首页；首页允许更换学段，但更换不影响连续打卡天数。

## 路由与守卫
- 新增页面：`/onboarding`（登录后可访问）。
- 新增守卫：`LevelGuard`（无学段时将用户重定向到`/onboarding`）。
- 路由调整：
  - `/login` → 登录页。
  - `/onboarding` → 新用户流程页（使用现有 `Onboarding` 组件）。
  - `/` → 首页（`AuthGuard` + `LevelGuard` 包裹，确保无学段时不会渲染首页）。
  - `/learn/:level`、`/review` → 保持 `AuthGuard`，不需 `LevelGuard`（学习页会按 URL 校验）。
- 登录跳转策略：保持现有“返回来源路径”的逻辑，同时由 `LevelGuard`决定是否进入`/onboarding`，避免登录后的竞态问题。

## 前端实现
- 新增 `pages/OnboardingPage.tsx`：
  - 渲染 `Onboarding`，在选择学段时调用 `PATCH /api/me` 写入 `educationLevel`，记录本地 `levelSince`，完成后 `navigate('/')`。
- 调整 `HomePage`：
  - 去掉首登的 `Onboarding` 分支，始终渲染仪表盘视图（开始学习、连续打卡、掌握单词与“更换学段”入口）。
- 新增 `LevelGuard.tsx`：
  - 从 `AppContext` 读取服务端的 `educationLevel`（已实现），不存在则重定向到`/onboarding`。
- 取词与展示：
  - 继续按当前实现使用 `levelSince` 过滤服务端掌握列表，避免重复推荐；首页掌握数量使用 `GET /api/learning/mastery/count?since=...`。

## 后端与服务
- 保持已扩展的 `GET /api/me` 返回 `educationLevel`；使用已新增的 `GET /api/learning/mastery/count`。
- 无需进一步后端变更即可满足“新用户流程”路由需求。

## 验证
- 首次注册/登录 → 自动重定向到`/onboarding` → 选择学段 → 进入首页仪表盘。
- 再次登录 → 直接进入首页仪表盘；更换学段后 streak 不变，掌握数量从新学段起始时间统计。
- 学习页与复习页按当前学段工作正常。
