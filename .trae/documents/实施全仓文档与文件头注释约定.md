## 目标
- 在根 README 明确“更新必须同步文档”的约定。
- 每个目录新增/刷新极简架构说明（≤3行）与文件清单（文件名、地位、功能），并声明目录变更需同步此文档。
- 每个代码文件开头加入三行极简注释：input / output / pos；并附更新提醒行。

## 覆盖范围
- 根目录：`README.md` 增加“文档更新约定”小节。
- 需要目录文档的目录（示例）：
  - `backend/`、`backend/src/`、`backend/src/common/`、`backend/src/modules/`、`backend/src/modules/{auth,learning,stats,user}/`
  - `frontend/`、`frontend/components/`、`frontend/contexts/`、`frontend/pages/`、`frontend/router/`、`frontend/services/`
  - `deploy/`、`deploy/{api,backend,frontend,nginx}/`
  - `data/`、`data/vocab/`
- 需要文件头注释的类型：`*.ts`, `*.tsx`, `*.js`, `*.prisma`, `*.sql`, `*.yml`, `*.yaml`, `*.toml`, `Dockerfile`, `*.conf`
- 排除：`node_modules/`, `dist/`, `build/`, `.vercel/`, `*.md`, `*.json`, `package-lock.json` 等不适合加入头注释的文件。

## 根 README 更新（新增小节）
- 标题：`文档与更新约定`
- 内容要点：
  - 任意“功能/架构/写法”改动完成后，必须同步更新相关目录的 README 与受影响文件的头注释。
  - 未同步更新视为变更未完成，变更提交需包含文档同步。

## 目录 README 模板
- 置顶三行架构说明（严格 ≤3 行）：
  1. 本目录职责边界（例如：UI 原子组件/后端鉴权模块/部署脚本）
  2. 数据/依赖来源（例如：仅 props/context；或经 `PrismaService`/`JWT`）
  3. 与上层/下层的关系（例如：服务→控制器→路由；组件→页面）
- 紧随其后加入声明：`一旦我所属的文件夹有所变化，请更新我。`
- “文件清单”逐行：`文件名 — 地位 — 功能`
- 示例（`frontend/components/README.md`）：
  ```
  组件层：原子 UI 组件集合
  数据来源：仅 props/context，无直连 API
  关系：被页面组合使用
  一旦我所属的文件夹有所变化，请更新我。
  
  文件清单：
  - Auth.tsx — 组件 — 登录状态与按钮
  - Button.tsx — 组件 — 标准按钮
  - LoadingOverlay.tsx — 组件 — 全局加载遮罩
  ```

## 文件头注释模板
- 三行极简信息 + 更新提醒（第4行）：
  - `input`：外部依赖（库/服务/上下文）
  - `output`：对外导出（符号/路由/副作用）
  - `pos`：系统中的局部地位（层级/模块）
  - 更新提醒：`若我被更新，请同步更新我的开头注释，以及所属的文件夹的 README。`
- 语法按文件类型：
  - `ts/tsx/js`：`// ...`
  - `prisma`：`// ...`
  - `sql`：`-- ...`
  - `yml/yaml/toml/Dockerfile/conf`：`# ...`
- 示例（`frontend/components/Auth.tsx`）：
  ```ts
  // input: React, AppContext, apiClient
  // output: <Auth/> 组件（导出 AuthProps）
  // pos: 前端/组件层—认证与按钮状态
  // 若我被更新，请同步更新我的开头注释，以及所属的文件夹的 README。
  ```

## 自动化实施
- 在根新增脚本 `tools/enforce-docs.ts`：
  - 扫描仓库，按上述“覆盖范围/排除”生成或校验目录 README 与文件头注释。
  - 解析 `import/export` 自动填充：
    - `input`：收集外部包与本地模块引入（如 `react`, `apiClient`, `PrismaService`）。
    - `output`：收集导出的类/函数/常量或推断控制器路由（Nest `@Controller`）。
    - `pos`：按路径与命名规则映射（如 `backend/src/modules/auth/*` → “后端/鉴权模块”）。
  - 生成 README 的“文件清单”部分：基于文件名模式（如 `*.controller.ts` → 控制器）。
  - 为不可注释的文件（如 `*.json`）在所属目录 README 添加“数据文件”说明而非修改文件本身。
- 在根 `package.json` 添加脚本：
  - `docs:enforce`：生成/更新全部文档与头注释。
  - `docs:check`：仅校验并在差异时退出非零。
- 可选：集成 Husky pre-commit，执行 `npm run docs:check`，阻止未同步文档的提交。

## 执行步骤
1. 运行 `docs:enforce` 生成首版目录 README 与文件头注释（保留可编辑占位）。
2. 人工查看并润色每个目录的三行架构说明与文件地位/功能，提交。
3. 更新根 README 加入“文档与更新约定”小节，提交。
4. 配置 `docs:check` 至 CI 或 pre-commit，确保后续变更必须同步文档。

## 验证与守护
- 校验规则：
  - 每个目标目录必须存在 README，且包含三行说明+更新声明+文件清单。
  - 每个目标文件必须存在 3 行头注释与更新提醒；语法与类型匹配。
- 提供报告：列出缺失/过期文档与文件，提示一键修复或手动编辑。

## 变更影响与注意
- 不改动业务逻辑，仅补充文档与注释。
- `*.json` 等不支持注释的文件通过目录 README 维护“输入/输出/地位”。
- 排除生成到第三方与构建产物目录，确保脚本安全可重复运行。

如确认以上方案，我将按照该计划实现脚本、批量生成文档与头注释，并提交首版供你快速检查。
