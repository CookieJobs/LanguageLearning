## 问题定位
- 前端调用接口使用相对路径 `"/api/..."`（frontend/services/geminiService.ts:4、18、57、77、99、106、112、123、129、134），开发环境通过 Vite 代理到本地后端（frontend/vite.config.ts:12）。
- 线上当前仅前端在 Vercel 上可访问，后端未对外暴露，因此 `/api/auth/login` 与 `/api/auth/register` 请求在生产环境无法到达后端。
- 后端实际路由：
  - `POST /api/auth/register`、`POST /api/auth/login`（backend/src/modules/auth/auth.controller.ts:13-15）
  - `GET /api/me`、`PATCH /api/me`（backend/src/modules/user/user.controller.ts:9-17）
  - `GET /api/stats/me`、`POST /api/stats/checkin`（backend/src/modules/stats/stats.controller.ts:7-13）
  - `POST /api/learning/*`（backend/src/modules/learning/learning.controller.ts:15-17, 24-26, 28-32, 35-42）
- 后端全局前缀与 CORS 已开启（backend/src/main.ts:12-15），健康检查为 `GET /api/health`（backend/src/modules/health.controller.ts:5）。

## 可选方案
- 方案 A：使用仓库提供的 Docker + NGINX 一体化部署（deploy/docker-compose.yml:1-28，deploy/nginx/nginx.conf:14-37）。
  - 优点：前后端同域，`/api` 由 NGINX 反代到后端，前端静态资源同站点部署，最少改动前端代码。
  - 缺点：需要自备服务器与域名/TLS。
- 方案 B：后端部署到 Render/Railway/Fly.io（或其他平台）获取独立域名，前端继续在 Vercel。
  - 通过 `vercel.json` 为 `/api/(.*)` 添加重写到后端域名，保持前端相对路径不变。
  - 优点：快速上线，无需维护服务器；缺点：跨平台配置与网络链路稍复杂。
- 建议优先采用方案 B 以尽快对外可用；若后续需要同域、稳定的带宽与更佳控制力，再升级到方案 A。

## 实施步骤（推荐 B）
### 部署后端
- 选择平台并创建服务（Node 20 运行时）。
- 设置环境变量（backend/.env.example 里的字段）：
  - `API_PORT=5500`
  - `DATABASE_URL`（生产建议使用 Postgres 托管库；若临时使用 SQLite，需挂载持久化卷）
  - `JWT_SECRET`（必须为强随机，避免使用默认 `dev_secret`，见 backend/src/modules/auth/auth.service.ts:12、39）
  - `DEEPSEEK_API_KEY`、`DEEPSEEK_MODEL=deepseek-chat`
- 构建与启动：参考 `deploy/backend/Dockerfile` 已包含 `npx prisma migrate deploy`（deploy/backend/Dockerfile:15）。非容器平台请在启动前执行：
  - `npm ci`
  - `npx prisma generate`
  - `npx prisma migrate deploy`
  - 启动应用（backend/src/main.ts:15 监听 `API_PORT`）。
- 验证健康检查：`GET https://<后端域名>/api/health` 应返回 `{ok:true}`。

### 连接前端与后端
- 在 Vercel 项目根添加 `/api/(.*)` 到后端的重写（不修改前端代码）：
  - `vercel.json` 样例：
    ```json
    {
      "installCommand": "npm ci --prefix frontend",
      "buildCommand": "npm run build:frontend",
      "outputDirectory": "frontend/dist",
      "rewrites": [
        { "source": "/api/(.*)", "destination": "https://<后端域名>/api/$1" },
        { "source": "/(.*)", "destination": "/index.html" }
      ]
    }
    ```
- 可选：如果更偏好显式配置 API 基址，前端支持改造成环境变量：在代码里以 `const base = import.meta.env.VITE_API_BASE_URL || ''` 统一前缀，再把所有 `fetch('/api/...')` 改为 `fetch(base + '/api/...')`，并在 Vercel 环境变量中设置 `VITE_API_BASE_URL=https://<后端域名>`。（此改动需要更新若干服务文件，当前计划先走重写，更少改动。）

### CORS 与安全
- 后端已 `app.enableCors({ origin: true, credentials: true })`（backend/src/main.ts:13），会回显请求来源；若平台有额外的 CORS 限制，请将前端域名加入白名单。
- 设置强随机 `JWT_SECRET`，并在泄露或变更时强制注销刷新令牌（AuthService 会在登录/刷新时存储刷新令牌哈希，backend/src/modules/auth/auth.service.ts:51）。
- 生产环境不要把 API Key 暴露到前端，后端服务调用第三方模型（DeepSeek）已在服务端完成（backend/src/modules/learning/deepseek.service.ts）。

### 会话与体验优化（后续增强）
- 自动刷新令牌：当前前端只在 401 时抛出事件（frontend/services/apiClient.ts:19-21），可新增刷新逻辑：
  - 捕获 401 → 调用 `POST /api/auth/refresh` → 成功则重试原请求；失败则触发退出。
- 登录态初始化：`AppContext` 启动时调用 `getMe`（frontend/contexts/AppContext.tsx:67-79）；确保部署后该请求通畅，以显示用户信息与学习进度。

### 监控与可观测性
- 接入平台日志（后端）与前端错误上报（例如 Sentry）。
- 关键探针：`/api/health`、登录/注册成功率、`DeepSeek` 调用失败率。

### 域名与公开访问
- 为前端绑定自定义域名与 HTTPS（Vercel 提供一键 DNS 与证书）。
- 为后端绑定独立域名或平台提供的 HTTPS 地址；如采用方案 A，同域通过 NGINX 反代（deploy/nginx/nginx.conf:26-33）。
